EP78: 코드 리뷰 문화와 소프트웨어 설계 원칙

개요

이 노트는 EP78 팟캐스트에서 논의된 코드 리뷰 문화와 소프트웨어 설계 원칙에 대한 핵심 내용을 정리한다. 표면적으로는 코드 리뷰 프로세스 개선에 초점을 맞추고 있지만, 진정한 주제는 ‘인지 부하(Cognitive Load) 관리’와 ‘조직의 심리적 안전망 구축’ 이다.

핵심 통찰: 코드 리뷰의 본질은 ‘인지 부하 분산’

  • 표면적 문제: 코드 리뷰가 느리다, 리뷰어가 피곤하다, 리뷰가 형식적이다.
  • 숨겨진 의도: 리뷰어의 인지적 과부하를 줄이고, 리뷰이가 방어적 코딩에서 벗어나게 하는 것.
  • 전문가 통찰: 코드 리뷰는 단순한 ‘버그 찾기’가 아니라, 팀 전체의 사고(Thinking)를 동기화하는 의식(Ritual) 이다. 좋은 리뷰는 코드의 정확성보다 설계의 일관성과 변경 가능성을 평가한다.

주요 개념

1. 리뷰 단위의 크기와 인지 부하

  • 작은 PR(50줄 이하): 리뷰어의 작업 기억(Working Memory)에 부담이 적어, 표면적인 문법 오류뿐 아니라 아키텍처적 일관성까지 검토 가능.
  • 큰 PR(200줄 이상): 리뷰어가 작업 기억을 초과하여, 리뷰는 ‘LGTM’ 또는 ‘여기서 이렇게 했으면 좋겠어요’ 같은 모호한 피드백으로 변질됨.
  • 핵심: 리뷰 단위는 ‘한 번에 이해할 수 있는 논리적 단위(One Logical Change)’ 여야 함. 이는 DDD의 Aggregate 경계와 유사함.

2. 리뷰의 방어적 vs. 건설적 문화

  • 방어적 리뷰 문화:
    • 리뷰어가 “왜 이렇게 했나요?”라는 질문을 많이 던짐.
    • 리뷰이가 모든 결정을 정당화해야 한다는 압박을 느낌.
    • 결과: 리뷰어는 경계심을, 리뷰이는 소진(Burnout) 을 경험.
  • 건설적 리뷰 문화:
    • 리뷰어가 “이 부분이 나중에 X라는 변경이 있을 때 영향을 받을 수 있을 것 같아요. 대신 Y는 어떨까요?”라는 식으로 미래의 변경 가능성을 논의.
    • 핵심 동작: 질문이 아닌 제안(Proposal)을 던짐.
  • 통찰: 건설적 리뷰는 심리적 안전(Psychological Safety) 을 기반으로 함. 리뷰이가 실수를 두려워하지 않을 때, 더 혁신적인 설계가 나옴.

3. 리뷰 도구의 함정

  • GitHub PR 기반 리뷰의 한계:
    • diff 단위로 리뷰가 이루어져, 변경의 맥락(Context) 을 파악하기 어려움.
    • 리뷰어가 전체 파일 구조를 보지 못하고, 변경된 라인만 봄.
    • 결과: ‘나무만 보고 숲을 못 보는’ 상황 발생.
  • 해결 방안:
    • 리뷰 전에 설계 문서(ADR) 또는 커밋 메시지의 의도를 먼저 공유.
    • 리뷰어가 먼저 변경의 목적을 이해한 후, 코드를 검토하도록 프로세스 변경.
    • Pair Review(페어 리뷰): 두 명이 함께 리뷰하며, 인지 부하를 분산.

4. 리뷰와 설계 원칙의 연결

  • SOLID 원칙과 리뷰:
    • 단일 책임 원칙(SRP): 하나의 PR이 하나의 책임만 가져야 함. 여러 책임이 섞이면 리뷰 난이도 급상승.
    • 개방-폐쇄 원칙(OCP): 리뷰 시 “이 변경이 기존 코드를 수정하지 않고 확장 가능한가?”를 질문.
    • 의존성 역전 원칙(DIP): 리뷰어가 “이 모듈이 구체적인 구현이 아닌 추상화에 의존하고 있는가?”를 확인.
  • DDD와 리뷰:
    • Bounded Context 경계: 리뷰 시 서로 다른 Bounded Context 간의 의존성이 생기지 않았는지 확인.
    • Aggregate 경계: 하나의 PR이 하나의 Aggregate를 넘어서지 않아야 함.

숨겨진 연결고리: 리뷰 문화는 곧 아키텍처 품질

  • 표면적: 코드 리뷰는 품질 관리 도구.
  • 중간층: 코드 리뷰는 지식 전파와 팀 표준화 도구.
  • 깊은 층: 코드 리뷰는 조직의 인지 구조(Cognitive Architecture)를 반영하는 거울.
    • 리뷰가 느리다 → 시스템의 모듈화가 잘 안 되어 있다.
    • 리뷰에서 같은 논쟁이 반복된다 → 설계 원칙이 명확하지 않다.
    • 리뷰어가 특정 부분만 집중해서 본다 → 해당 부분의 복잡도가 과도하다.

결론

코드 리뷰 문화를 개선하려면, 리뷰 프로세스 자체를 바꾸기보다 시스템의 인지적 복잡도(Cognitive Complexity)를 낮추는 설계가 선행되어야 한다. 리뷰는 증상이고, 원인은 설계에 있다.
