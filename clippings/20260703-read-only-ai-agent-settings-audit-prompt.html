Tweet Content

역할: 당신은 세계 최고 수준의 PC/AI 에이전트 설정 감사 전문가입니다.

내 PC 환경을 읽기 전용으로만 진단하고, 최적화 플랜을 제시하세요. 파일 생성, 파일 수정, 삭제, 이동, 명령어 실행, 설정 변경은 절대 하지 않습니다. 승인 전에는 어떤 변경 작업도 하지 말고, 오직 구조 확인과 분석만 수행하세요.

진단 범위 확인

먼저 진단할 범위를 확인하세요.

예:

  • 홈 폴더 전체 ~/
  • 개발 작업 폴더만
  • 특정 프로젝트 폴더만
  • Claude Code / Codex / Cursor / Hermes Agent 설정 폴더만

범위가 불명확하면 바로 작업하지 말고 먼저 질문하세요.

진단 대상

1. 폴더 구조와 프로젝트 정리 상태

현재 작업 폴더를 기준으로, 하위 폴더 구조를 진단하세요.

확인할 항목:

  • 프로젝트 폴더, 문서 폴더, 다운로드 파일이 뒤섞여 있는지
  • 폴더 이름 규칙이 일관적인지
  • 너무 깊거나 너무 얕은 폴더 구조는 없는지
  • 중복되거나 오래 방치된 폴더는 없는지
  • 프로젝트들이 일관된 위치에 정리되어 있는지
  • 현재 구조가 유지보수와 백업에 적합한지

주의: 파일 내용은 함부로 열지 마세요. 구조, 이름, 위치, 계층만 진단하세요.

2. CLAUDE.md 및 에이전트 지침 문서

다음 문서들을 확인하세요.

  • 사용자 전체 설정: ~/.claude/CLAUDE.md
  • 각 프로젝트 폴더의 CLAUDE.md
  • 다른 에이전트의 agent.md, AGENTS.md, instructions.md 등

확인할 항목:

  • 내용이 너무 길거나 토큰을 많이 쓰는지
  • 중복 지시가 많은지
  • 항상 읽을 필요 없는 절차가 섞여 있는지
  • 프로젝트별 지침과 전체 지침이 잘 분리되어 있는지
  • Claude Code, Codex, Cursor, Hermes Agent 등이 함께 일할 때 충돌할 내용은 없는지
  • 병렬 작업이나 역할 분담에 방해되는 지침은 없는지

3. .claude/rules/

확인할 항목:

  • 규칙 파일이 적절한 위치에 있는지
  • 적용 범위가 너무 넓거나 좁지 않은지
  • 프로젝트별 규칙과 공통 규칙이 잘 나뉘어 있는지

4. Claude Code 스킬

대상:

  • .claude/skills/*/SKILL.md
  • 예전 방식의 .claude/commands/

확인할 항목:

  • 스킬 설명이 실제 사용 상황과 잘 맞는지
  • 어떤 말이 들어오면 스킬이 실행될지 명확한지
  • 본문이 너무 길거나 세부 지시가 과한지
  • 비슷한 스킬이 중복되어 있는지
  • 필요한데 잘 호출되지 않을 위험이 있는지
  • 예전 command 방식에서 skill 방식으로 옮기다 만 흔적이 있는지

5. 서브 에이전트

대상:

  • .claude/agents/

확인할 항목:

  • 역할 분담이 명확한지
  • 각 에이전트가 너무 많은 일을 맡고 있지 않은지
  • 읽기 전용으로 제한해야 할 에이전트가 쓰기 권한을 갖고 있지 않은지
  • tools 설정이 과도하지 않은지
  • model 지정이 적절한지
  • memory 사용 방식이 적절한지

6. 훅과 자동 실행 설정

대상:

  • settings.json
  • PreToolUse, PostToolUse 등 hook 설정

확인할 항목:

  • 위험한 명령 실행을 막는 가드레일이 있는지
  • 포매터, 린터, 알림이 자동화되어 있는지
  • 자동 실행이 너무 많아 작업 속도를 방해하지 않는지
  • 읽기 전용 감사 상황에서 위험한 훅이 작동할 가능성은 없는지

7. output-styles

확인할 항목:

  • 자주 쓰는 답변 스타일이 정리되어 있는지
  • 매번 반복해서 말하는 지시를 output-style로 옮길 수 있는지
  • 프로젝트별 output-style과 전체 output-style이 충돌하지 않는지

8. MCP 설정

대상:

  • .mcp.json
  • Claude/Cursor/Codex/Hermes 관련 MCP 설정

확인할 항목:

  • 사용하지 않는 MCP 서버가 남아 있는지
  • 같은 기능을 하는 서버가 중복되어 있는지
  • 권한이 과도하게 넓은 서버가 있는지
  • 프로젝트별로 필요한 MCP와 전체 설정에 둘 MCP가 잘 분리되어 있는지

9. settings.json과 permissions

확인 대상:

  • 사용자 전체 설정
  • 프로젝트 설정
  • 로컬 설정
  • 관리용 설정

확인할 항목:

  • 허용, 거부, 질문 권한이 적절히 분리되어 있는지
  • 프로젝트별로만 필요한 권한이 전체 설정에 들어가 있지 않은지
  • 쓰기, 삭제, 실행 권한이 과도하게 열려 있지 않은지
  • 읽기 전용 진단용 모드가 가능한지

10. 플러그인/마켓플레이스화 가능성

확인할 항목:

  • 반복해서 쓰는 설정을 플러그인처럼 재사용할 수 있는지
  • 여러 프로젝트에서 공통으로 쓰는 규칙을 분리할 수 있는지
  • 개인 전용 템플릿으로 만들 만한 설정이 있는지

11. 모델 운영 전략

확인할 항목:

  • 작업 종류별로 모델을 구분해서 쓰고 있는지
  • 예: Fable / Opus / Sonnet / Haiku
  • 작업 난이도에 따라 thinking/effort 지침을 나눌 수 있는지
  • 가벼운 작업에 너무 비싼 모델을 쓰고 있지 않은지
  • 무거운 작업을 작은 모델에게 맡겨 품질이 떨어지는 문제는 없는지

12. 컨텍스트 효율

확인할 항목:

  • /compact를 적절히 쓰고 있는지
  • 항상 읽히는 문서가 너무 길지 않은지
  • 무거운 분석은 메인 대화가 아니라 서브 에이전트로 넘길 수 있는지
  • 매번 반복하는 지시를 어디로 옮기면 좋은지
  • CLAUDE.md, 스킬, rules, output-style의 역할이 잘 분리되어 있는지

13. 자동화 상태

확인할 항목:

  • 반복 작업을 스케줄 실행할 수 있는지
  • CLI 명령 하나로 실행 가능한 작업이 있는지
  • GitHub Actions로 자동화할 수 있는 작업이 있는지
  • pre-commit으로 자동 검사할 수 있는 작업이 있는지
  • 수동으로 반복하는 일을 스킬, 훅, 서브 에이전트로 바꿀 수 있는지

14. Git과 작업 흐름

확인할 항목:

  • 커밋 규칙이 있는지
  • 브랜치 전략이 있는지
  • 리뷰와 승인 흐름이 있는지
  • AI 에이전트가 직접 수정하기 전 승인 단계를 거치도록 되어 있는지
  • Codex, Claude Code, Cursor가 같은 프로젝트를 만질 때 충돌을 줄이는 구조인지

일상 사용 방식 재검토

현재 사용 방식에서 다음을 찾아내세요.

  • 더 빠르고 효율적으로 만들 수 있는 부분
  • 병렬화하면 좋은 작업
  • 자동화하면 좋은 반복 작업
  • 컨텍스트를 아낄 수 있는 부분
  • 스킬로 만들면 좋은 작업
  • 훅으로 자동화하면 좋은 작업
  • 서브 에이전트에게 넘기면 좋은 작업
  • 매번 말하는 지시를 CLAUDE.md나 output-style로 옮기면 좋은 부분

결과물 형식

진단 결과는 아래 순서로 정리하세요.

  1. 스캔 범위 확인
  2. 현황 맵
  3. 문제점 우선순위
  4. 이상적인 최적 구성안
  5. 단계별 실행 플랜

각 실행 단계에는 목표, 구체적인 작업, 예상 효과, 위험도, 되돌리는 방법, 사용자 승인이 필요한 작업을 포함하세요. 특히 파일 이동, 삭제, 설정 변경처럼 되돌리기 어려운 작업은 반드시 별도 단계로 분리하고, 실행 전에 반드시 승인을 받도록 표시하세요.
